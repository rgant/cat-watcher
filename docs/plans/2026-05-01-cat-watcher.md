# Cat Watcher — Version 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:subagent-driven-development (recommended) or
> superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the Version 1 Cat Watcher system per
`docs/specs/2026-05-01-cat-watcher-design.md`: ingest motion clips from two
Amcrest cameras, classify cat-vs-no-cat, browse via web UI, alert on inactivity
/ frequency / agent failures, run unattended on a Mac mini.

**Architecture:** Single `cat_watcher` Python package, four LaunchAgents
(poller, alerts, web, backup) sharing one SQLite DB on internal storage and
clips/thumbs/backups on an external drive. Strict file-before-DB-row ordering;
preservation semantics on camera state; cross-agent heartbeat watchdog;
cool-down keyed on `(camera_id, alert_type)` for camera alerts and
`(NULL, alert_type)` for the seven non-camera alert types.

**Tech Stack:** Python 3.14, pixi (conda-forge for arch-pinned ML wheels + PyPI
for everything else), FastAPI + Jinja2 + htmx, SQLAlchemy 2.0 + Alembic, SQLite
(WAL), pydantic-settings, ultralytics YOLOv11n, ffmpeg, httpx,
ruff/basedpyright/mypy/pylint, AGPL-3.0.

> [!TIP]
> This project uses python 3.14. It does not need
> `from __future__ import annotations`

## How to Use This Plan

1. **Sequence is strict.** Tasks build on each other; do not reorder.
2. **Test Driven Development.** Use TDD best practices.
3. **Commit policy.** The user uses **signed git commits** (per CLAUDE.md). The
   implementing agent must NOT run `git commit` directly. Each task ends with a
   "Commit checkpoint" step that prepares the working tree (`git status` clean
   of untracked work, all tests passing) and tells the user the suggested commit
   message — the user runs the commit themselves.
4. **Dependency policy.** Never edit `pyproject.toml` to add/remove/upgrade
   dependencies by hand. Use the project's package manager CLI (per CLAUDE.md).
   The manager is `pixi`. PyPI deps are added with `pixi add --pypi <pkg>`
   (runtime) or `pixi add --pypi --feature dev <pkg>` (dev). Conda-forge deps
   (the arch-pinned ML stack) are added with `pixi add <pkg>` — no `--pypi`
   flag. Use `pixi remove [--pypi] [--feature dev] <pkg>` to drop. Never edit
   `pyproject.toml` dependency blocks by hand.
5. **Lint policy.** Linters are project standards. If a rule fires, the default
   is to comply, not silence (per CLAUDE.md). Silencing requires explicit user
   approval.
6. **Test doubles policy.** Per CLAUDE.md: no double > fake > stub > spy > mock.
   Mock only at boundaries owned by third parties (Amcrest API via respx, SMTP
   via aiosmtpd or `unittest.mock.patch('smtplib.SMTP')`, `osascript` via
   `subprocess.run` patch, YOLO model via
   `unittest.mock.patch('ultralytics.YOLO')`). Internal `cat_watcher.*` modules
   are real in tests.
7. **Phase checkpoints.** The plan is divided into 7 phases. After each phase,
   all tests pass and the system is in a consistent state — a natural place to
   pause for review.
8. **Reference inputs.** The `tmp-env-template/` directory contains tooling
   files copied from a sibling project. Reuse them verbatim where the spec says
   so (§10.2). The directory is deleted in Task 28.

## Resolutions for spec §13 open items

The spec defers five decisions to writing-plans. This plan resolves them as
follows:

1. **`amcrest_client` implementation** — `httpx.Client` with `httpx.DigestAuth`,
   not `python-amcrest`. Reasoning: (a) `python-amcrest` is licensed
   **GPL-2.0-only**, which is incompatible with the project's
   `AGPL-3.0-or-later` license per FSF compatibility rules; (b) it pulls
   `requests` + `urllib3` + `argcomplete` + sync/async stacks alongside `httpx`,
   which is heavy for a 2-camera 5-min cadence; (c) it does not yet declare
   Python 3.14 support (classifiers stop at 3.11). The API surface we need
   (`factory.create` → `findFile` → `findNextFile` + streaming download) is
   small enough to implement directly against the vendored
   `Amcrest-HTTP_API_V3.26.pdf`. (Task 14.)
2. **First-run model-cache strategy** — **eager** via the
   `cat-watcher fetch-models` sub-command. The operator runs it during install,
   before `pixi run install-agents`, so the first poller tick is offline-safe.
   (Task 25.)
3. **Initial Alembic migration** — autogenerated from the Task 10 models, then
   reviewed by hand. (Task 11.)
4. **Test fixture video source** — synthesized at test time via ffmpeg's
   `lavfi testsrc` filter; no bundled image fixture, no third-party attribution.
   (Task 12. The original plan bundled a Wikimedia calico-kitten JPEG; Task 15
   made the image content irrelevant to every test, so the JPEG and its CC BY
   2.0 attribution were removed.)
5. **Cool-down key for non-camera alerts** —
   `(camera_id IS NULL,
   alert_type)`: one global cool-down per non-camera
   alert type. Applies to `POLLER_STUCK`, `WEB_DOWN`, `WEB_FLAPPING`,
   `ALERTS_STUCK`, `DISK_LOW`, `STORAGE_UNAVAILABLE`, `BACKUP_STALE`. (Task 18.)
6. **Camera local timezone (UTC conversion source)** — add an optional
   `cameras[].timezone` field to `config.toml`; if unset, fall back to
   `web.display_timezone` (the common case is one operator with all cameras and
   the host in the same zone). Final verification of which Amcrest API endpoint
   reports the camera's configured TZ happens during Task 14 implementation (the
   `Amcrest-HTTP_API_V3.26.pdf` is the reference); `pixi run test-cameras` then
   reports any drift between camera-reported and configured TZ at install time,
   and the operator sets `cameras[].timezone` explicitly if mismatched. (Tasks
   8, 14, 25.)

### Task 7a: Migrate from uv to pixi — retrospective

**Why:** PyPI dropped macOS-x86_64 wheels for the ML stack (torch, onnxruntime,
likely opencv-python in upcoming releases). Pixi sources from conda-forge, which
maintains osx-64 builds. This is the only workable path that doesn't require
hardware replacement or Docker on the Mac mini.

**What landed (deviates from spec §10.3 — keep this section as the source of
truth for the dependency split):**

- **Conda-forge** (in `[tool.pixi.dependencies]`) — only the packages with arch
  concerns: `python = "3.14.*"`, `pytorch`, `ultralytics`, `opencv`, `numpy`,
  `ffmpeg`. (Spec §10.3 also lists `pillow`; in practice it is pulled
  transitively and does not need to be pinned at the project root.)
- **PyPI runtime** (in `[project] dependencies`) — `alembic`, `fastapi`,
  `httpx`, `jinja2`, `pydantic-settings`, `python-dotenv`, `sqlalchemy`,
  `uvicorn[standard]`. Standard PEP 621 location.
- **PyPI dev** (in `[dependency-groups] dev`, PEP 735) — `aiosmtpd`,
  `basedpyright`, `mypy`, `pylint`, `pylint-per-file-ignores`,
  `pylint-pytest>=2.0.0a1`, `pylint-venv`, `pyproject-fmt`, `pytest`,
  `pytest-cov`, `pytest-mock`, `pytest-randomly`, `respx`, `ruff`. Pixi
  auto-maps the `dev` group to a feature; the `default` environment opts in.
- **Pixi-only PyPI block** (in `[tool.pixi.pypi-dependencies]`) — just the
  editable self-install: `cat-watcher = { path = ".", editable = true }`.
- **`[tool.pixi.workspace]`** — `channels = ["conda-forge"]`,
  `platforms = ["linux-64", "osx-64", "osx-arm64"]`.
- **`[tool.pixi.environments]`** —
  `default = { features = ["dev"],
  solve-group = "default" }`.

The split rule: PyPI for everything except packages whose macOS-x86_64 wheels
are missing on PyPI but available on conda-forge. Standard PEP 621 / PEP 735
locations are preferred over `[tool.pixi.feature.*.dependencies]` because pixi
already understands them and IDE / build tooling reads them natively.
**Therefore the `[project] dependencies` and `[dependency-groups]` blocks stay**
— they are the canonical source for everything the prod machine can fetch from
PyPI directly.

**Brewfile:** added `brew "pixi"`, removed `brew "uv"`. (`uv` was only useful
during the migration window and is gone now that the runtime is fully pixi.)

**Task runner:** `[tool.pixi.tasks]` replaces the `justfile`. All pixi tasks
listed in spec §10.4 are in place; `pixi task list` shows the full surface.

**`.gitignore`:** `.pixi/` excluded (per-machine cache). `pixi.lock` tracked.

### Task 9: Storage module — DONE

**Status:** core + test refinement (Task 9b) landed. 16 tests, 100% coverage on
`src/cat_watcher/storage.py`. All four linters clean.

**Files:**

- Create: `src/cat_watcher/storage.py`
- Create: `tests/unit/test_storage.py`

**Implementation Requirements:**

- Implement layout management to ensure predictable persistence directories.
- Define exceptions: `StorageError` (base) and `StorageUnavailableError`
  (subclass) raised when a wait times out.
- `ensure_storage_layout(internal_root, storage_root)` generates required
  subfolders for internal operations (`models`, `logs`) and storage operations
  (`clips`, `thumbs`, `backups`). Loudly fails via `StorageError` if the root
  paths themselves do not exist (operator fixed a typo, didn't auto-create a
  wrong directory).
- `storage_available(path) -> bool` — true if `path` is an existing, writable
  directory (verified by writing then deleting a `tempfile.NamedTemporaryFile`
  inside it).
- `wait_for_storage(path, *, interval_seconds, timeout_seconds) -> None` — EAFP.
  Polls `storage_available(path)` every `interval_seconds`. Returns `None` on
  success. Raises `StorageUnavailableError` when the cumulative wait reaches
  `timeout_seconds`. Both keyword args default to the values in `[storage]`
  config (`wait_interval_seconds = 10`, `wait_timeout_seconds = 600`); tests
  override.
- Callers: `try: wait_for_storage(...)` then proceed;
  `except
  StorageUnavailableError:` log CRITICAL and exit non-zero.

**Test Requirements (`tests/unit/test_storage.py`):**

- `ensure_storage_layout` correctly generates the exact internal and storage
  subfolders.
- `ensure_storage_layout` is idempotent on repeated executions.
- `ensure_storage_layout` raises `StorageError` if the root paths are missing.
- `storage_available` for both writable and missing directory conditions.
- `wait_for_storage` returns `None` immediately when the path is available.
- `wait_for_storage` raises `StorageUnavailableError` on timeout (mock
  `time.monotonic` and `time.sleep` so the test runs in milliseconds).

### Task 10: DB models + session — DONE

**Status:** core + test refinement (Task 10b) landed. 11 tests, 98% coverage on
`src/cat_watcher/db.py` (the 2% gap is the dialect-guard `ValueError` raise —
accepted; tested via boundary mock would only re-test SA's own behavior). All
four linters clean, pylint 10.00/10.

**Decision worth recording:** added a custom `UtcDateTime(TypeDecorator)` to
wrap every datetime column. Required because SQLAlchemy + SQLite silently drop
tzinfo on read, contradicting the spec's "all datetimes tz-aware UTC" contract.
The decorator rejects naive datetimes on bind (loud failure beats silent
timezone drift) and stamps UTC on results. Used on all 11 datetime columns
across the 5 models.

**Files:**

- Create: `src/cat_watcher/db.py`
- Create: `tests/unit/test_db.py`

**Implementation Requirements:**

- Establish an SQLAlchemy 2.0 foundation utilizing declarative mapping to
  interface with SQLite.
- Define an engine factory that executes PRAGMA statements upon connection to
  enforce Write-Ahead Logging (WAL) and foreign key constraints.
- Implement a transactional `get_session` context manager that commits on
  success and rolls back on exception.
- Define the following ORM models: `Camera`, `Clip`, `AlertSent`, `Heartbeat`,
  and `AgentStart`.
- Ensure the schema strictly enforces a unique constraint bridging `camera_id`
  and `source_filename` within the `Clip` model, and implements cascading
  deletions from `Camera` to `Clip` models.

**Test Requirements (`tests/unit/test_db.py`):**

- Test the engine factory to confirm that the SQLite pragmas for WAL mode and
  foreign keys are actively applied to new connections.
- Test the complete model round-trip by instantiating dependent records
  (`Camera`, `Clip`, `AlertSent`), flushing them through the session, and
  successfully retrieving them.
- Test the database integrity constraints by attempting to insert duplicate
  `Clip` records with identical camera IDs and source filenames, asserting that
  an `IntegrityError` is raised.

`docs/resources/amcrest-bracket-quirk.md`:

  typos) propagate. `CameraAPIError.status: int | None` exposes the HTTP status
  code so callers discriminate structurally rather than parsing the message

### Task 16: Retention Sweep — DONE

**Status:** core landed. 14 unit tests, 92% coverage (the misses are
race-condition guards against rows vanishing between SELECT and per-row DELETE).
All four linters clean, pylint 10.00/10.

**Implementation surface:**

- `sweep(*, engine, storage_root, retention, now) -> RetentionReport` — one
  entry point that runs all five sub-passes (DB-driven clip removal, orphan
  filesystem sweep, empty date-directory cleanup, `agent_starts` prune,
  `alerts_sent` prune) and returns a frozen `RetentionReport` with per-pass
  counts.
- **Pass 1:** queries aged Clip IDs, then for each ID opens its own transaction
  and deletes the row before unlinking the files. Row-first ordering guarantees
  a crash leaves only an orphan file (handled by pass 2), never a dangling DB
  row.
- **Pass 2:** loads the surviving `Clip.file_path` + `Clip.thumb_path` set once,
  walks `<storage_root>/clips` and `<storage_root>/thumbs` via `rglob`, and
  unlinks files whose mtime is past the cutoff and whose relative path is not in
  the survivor set. O(N + M).
- **Empty date-dir cleanup:** `rmdir` each `<top>/<slug>/<YYYY-MM-DD>/`
  directory; raises on non-empty (intentional — caught and skipped).
- **`agent_starts` / `alerts_sent` pruning:** independent `DELETE WHERE` per
  table using the corresponding cutoff from `RetentionConfig`.
- **Resilience:** every filesystem operation is wrapped — unlink failures and
  stat failures log WARNING and continue; the sweep never raises through to the
  caller.

**Files:**

- `src/cat_watcher/retention.py`
- `tests/unit/test_retention.py`

#### Implementation Requirements

- **Data Purge Logic:** `sweep(*, retention: RetentionConfig, now:
  datetime)`
  — durations come from `[retention]` config (`clip_days`, `agent_starts_days`,
  `alerts_sent_days`, all default 30 per Task 8). Tests pass an explicit `now`
  for determinism.
- **Two-pass clip sweep (per spec §4.4 step 7):** the clip portion of the sweep
  runs in two distinct passes that the test suite exercises independently:
  1. **DB-driven pass.** For each clip with
     `start_ts < now -
     retention.clip_days`: delete the row inside a
     transaction first, then unlink the clip and thumbnail files. Row-first
     ordering means a crash between the two leaves only an orphan file, never a
     dangling row pointing at a missing file.
  2. **Orphan sweep.** Scan `clips/` and `thumbs/` for files whose mtime is
     older than `now - retention.clip_days` and that have no matching `Clip`
     row, and unlink them. This cleans the orphans pass 1 leaves behind on
     crash, plus any other stray files (e.g., a partially-written download from
     before the file-before-row ordering existed).
     - **Efficiency note:** load the surviving `Clip.file_path` and
       `Clip.thumb_path` values into a single in-memory `set` once, then walk
       the filesystem and check membership. Don't issue one DB query per file —
       the sweep should be O(N + M), not O(N × M).
  3. **Empty date directory cleanup.** After both passes, walk `clips/<slug>/`
     and `thumbs/<slug>/`; remove any date subdirectory that is empty
     (`os.rmdir`). Otherwise empty `2024-01-15/` dirs accumulate forever even
     though the spec promised "delete entire date directories at once" (§6.1).
- **Other tables:**
  - Prune `agent_starts` rows older than `now - retention.agent_starts_days`
    (matches the visibility horizon the `WEB_FLAPPING` watchdog and the forensic
    "when-did-X-restart" UI rely on).
  - Prune `alerts_sent` rows older than `now - retention.alerts_sent_days`
    (matches `/alerts` route's "last N days" claim — the route reads the same
    config value).
- **Resilience:** Handle `OSError` exceptions during file deletion or directory
  cleanup (file already missing, dir not empty due to race) without crashing the
  sweep — log WARNING and continue.
- **Reporting:** Return a structured `RetentionReport` dataclass with counts:
  `clips_removed_pass1`, `orphans_removed_pass2`, `dirs_removed`,
  `agent_starts_pruned`, `alerts_sent_pruned`.

#### Test Requirements

- **Pass 1 (functional sweep):** Records and files older than the cutoff are
  removed; "younger" data remains intact.
- **Pass 2 (orphan sweep):** Pre-seed an orphan file (no matching Clip row) with
  old mtime; verify it is unlinked. Pre-seed a young orphan file; verify it is
  preserved.
- **Crash-tolerance shape:** Simulate a pass 1 crash between row delete and file
  unlink; verify the next sweep's pass 2 cleans up the orphan.
- **Empty-dir cleanup:** seed a date directory whose only files are removed by
  pass 1; verify the directory is `rmdir`-ed in pass 3. Seed a date directory
  with one young file remaining; verify the directory is kept.
- **Configurability:** with `retention.clip_days = 7`, files older than 7 days
  are removed; with the default 30, only files older than 30 days are removed.
- **Idempotency:** Running the sweep twice in a row results in zero additional
  deletions and no errors.
- **Integrity:** File deletion failures are logged as warnings but do not halt
  the process.
- **agent_starts / alerts_sent pruning:** Old rows in both tables are removed;
  recent rows are preserved. Independent retention configs (e.g.,
  `agent_starts_days = 7` with `alerts_sent_days = 30`) honored.

  - Iterate through available recordings since the last successful poll

  distinguish, add a separate `analysis_skipped: bool` column then.

#### Module split rationale

Three distinct concerns live in three places to keep imports one-way and
testability high:

- `cat_watcher.notifier` — pure outbound senders (Task 13). Knows nothing about
  alert types, cool-down, or DB state.
- `cat_watcher.alert_templates` — pure renderers. One function per alert type
  (`render_inactivity(camera, branch, ...) -> AlertContent`,
  `render_frequency(camera, count, ...)`, etc.). Returns an `AlertContent`
  dataclass with `subject`, `body`, `macos_summary` (≤120 chars). No I/O, no
  config, no DB. Trivially testable as snapshot tests.
- `cat_watcher.alerts` — rule evaluators + the shared `dispatch_alert(...)`
  helper. The poller imports `dispatch_alert` for `ALERTS_STUCK` (Task 17).
  Imports flow `poller → alerts → {notifier, alert_templates}`; nothing imports
  `poller`, so no circular import risk.

#### Shared dispatch helper

`dispatch_alert(alert_type: AlertType, *, camera_id: int | None,
content: AlertContent, secrets: EmailSecrets, rules: AlertConfig,
session: Session) -> None`

Behavior:

1. Compute the cool-down key: `(camera_id, alert_type)` for camera alerts;
   `(NULL, alert_type)` for the seven non-camera types.
2. Look up the most recent `alerts_sent` row matching the key. If
   `now - sent_at < cooldown_for(alert_type, rules)`, **log at INFO** with the
   suppression reason and return — **do NOT write a suppressed row to
   `alerts_sent`** (otherwise an active 24h-cool-down alert would generate ~96
   rows/day per type). Spec §6.2 has no `suppressed` column, by design.
3. Otherwise call `notifier.send_email(...)` and
   `notifier.send_macos_notification(...)` (both honor their `enabled` flags).
   Write a single `alerts_sent` row capturing `email_ok` / `macos_ok` /
   `delivery_error`.

`cooldown_for(alert_type, rules)` returns `_HARDCODED_COOLDOWN_OVERRIDES`'s
value if the type is in the dict (`WEB_FLAPPING=1h`, `DISK_LOW=24h`,
`BACKUP_STALE=24h`); otherwise `rules.cooldown_hours` (default 6h). The
overrides live in a module-level dict in `cat_watcher.alerts` with a comment
explaining why they are intentionally not in `config.toml` (per spec §4.5: a
flap detector with a 6h cool-down would miss the next flap; a once-per-24h
backup's staleness check would fire repeatedly within a single missed run if the
cool-down were hours-scale). One-line edit if the operator needs to change them;
no config schema churn.

#### 1. Plist Template Requirements

Each template must define a macOS LaunchAgent using the following configuration
surfaces to ensure consistent behavior and observability:

- **Execution Identity:** A unique `Label` following the
  `com.robgant.cat-watcher.[service]` convention.
- **Context:** Set `WorkingDirectory` to the repository root and provide the
  `CAT_WATCHER_CONFIG` environment variable.
- **Pathing:** Use `__REPO_DIR__` and `__INTERNAL_ROOT__` placeholders for
  dynamic substitution during installation.
- **Logging:** Redirect `StandardOutPath` and `StandardErrorPath` to
  `<internal_root>/logs/<agent>.{stdout,stderr}.log` (per spec §6.1) so logs
  remain accessible even when the external drive is offline.
- **`RunAtLoad`:** Every plist sets `<key>RunAtLoad</key><true/>`. Without this,
  periodic agents (`StartInterval`-based) do not fire on bootstrap until the
  next interval rolls over, which delays observability of any install issue.
- **Service Specifics — cadences read from `config.toml` at install time (the
  install script renders the templated plist with values from `[poller]`,
  `[alerts]`, `[backup]`):**
  - **Poller:**
    `<key>StartInterval</key><integer>{{ poller.cadence_seconds }}</integer>`
    (default 300 = 5 min).
  - **Alerts:**
    `<key>StartInterval</key><integer>{{ alerts.cadence_seconds }}</integer>`
    (default 900 = 15 min).
  - **Web:** Long-running process with `KeepAlive` (restart on non-zero exit)
    and `<key>ThrottleInterval</key><integer>30</integer>` to prevent
    crash-looping (per spec §8.2).
  - **Backup:** `StartCalendarInterval` with `Hour={{ backup.cadence_hour }}`
    and `Minute={{ backup.cadence_minute }}` (defaults 3, 0 — 03:00 local).

#### 2. Installation & Uninstallation Logic

The shell scripts manage the transition from templates to active system
services.

- **`install_launchagents.sh`:**
  - **Cadence injection from config:** invoke
    `pixi run python -m cat_watcher.scripts.render_plists` (a small helper) to
    read `config.toml` and emit the four final plist files into a temp dir. The
    helper does the templating in Python (where the config is already parsed)
    rather than asking the shell to parse TOML. The script then `cp`'s the
    rendered plists into `~/LaunchAgents/`.
  - **Placeholder injection:** `__REPO_DIR__` and `__INTERNAL_ROOT__`
    placeholders are replaced by the Python helper at the same time it writes
    the cadence values.
  - **Idempotency:** The script must handle existing agents by performing a
    `bootout` (if loaded) before bootstrapping the new plist.
  - **Directory Management:** Ensure the target log and `LaunchAgents`
    directories exist before deployment.

#### Implementation Requirements (`logging_setup.py`)

**`JsonFormatter(logging.Formatter)`:**

- Constructor: `JsonFormatter(*, agent_name: str)`. Stores `agent_name` as an
  attribute; `pid` is read via `os.getpid()` at format time.
- `format(record: logging.LogRecord) -> str` returns a single-line JSON string
  (no embedded newlines) matching the spec's schema:
  - Required keys: `ts`, `level`, `logger`, `agent`, `pid`, `msg`.
  - `ts`: ISO 8601 UTC with microseconds, e.g.
    `"2026-05-05T18:42:13.123456+00:00"`. Derive from `record.created`.
  - `msg`: the `%`-substituted formatted message (`record.getMessage()`).
  - Optional `extras`: present when the record carries non-standard attributes
    (i.e. anything passed via `extra={...}` or set by a `LoggerAdapter`).
    Compute by enumerating `record.__dict__` and excluding the standard
    `LogRecord` attribute names. If no non-standard attributes exist, omit the
    `extras` key entirely (do not emit `{}`).
  - On exceptions (`record.exc_info` not None or False): emit `exc_type`
    (fully-qualified class name), `exc_msg` (`str(exc)`), and `traceback` (a
    single string from `traceback.format_exception(...)`).
- The single-line invariant matters: if any field contains a newline, the
  formatter must escape it (`json.dumps(..., ensure_ascii=False)` already
  handles this for the `\n` literal; tracebacks come out as `\n`-separated
  strings inside the JSON, which is correct).
- The standard-attribute set used to compute `extras` must be defined as a
  module-level frozenset enumerating every attribute Python's `logging` module
  sets on a `LogRecord` (see `logging.LogRecord.__init__` — `name`, `msg`,
  `args`, `levelname`, `levelno`, `pathname`, `filename`, `module`, `exc_info`,
  `exc_text`, `stack_info`, `lineno`, `funcName`, `created`, `msecs`,
  `relativeCreated`, `thread`, `threadName`, `processName`, `process`,
  `message`, `taskName`).

**`setup_logging(*, agent_name: str, internal_root: Path, level: int) -> None`:**

- `(internal_root / "logs").mkdir(parents=True, exist_ok=True)` for safety
  (matches Task 9's `ensure_storage_layout` contract; tolerates being called
  from a script that hasn't run that yet).
- Construct
  `RotatingFileHandler(<internal_root>/logs/<agent_name>.jsonl,
  maxBytes=10 * 1024 * 1024, backupCount=7, encoding="utf-8")`.
  Both `maxBytes` and `backupCount` should live in module-level constants so
  tests can monkey-patch them.
- Construct `StreamHandler(sys.stderr)` with `level=logging.WARNING` so genuine
  problems still hit the LaunchAgent's `<agent>.stderr.log` fallback even when
  the JSONL file isn't being read.
- Both handlers are wired with `JsonFormatter(agent_name=agent_name)` so
  structured filtering works on either stream.
- Call sequence: clear existing handlers from the root logger first, then attach
  the two new handlers, then set the root logger's level to `level`. This makes
  `setup_logging` idempotent — calling it twice results in exactly two handlers
  attached, not four.
- Do NOT call `logging.basicConfig`. The replacement-not-augment behavior is the
  reason for the explicit `clear-then-attach` sequence.

#### Implementation Requirements (`cat-watcher logs` viewer)

CLI surface (registered as a sub-command on the umbrella parser in
`__main__.py`):

```text
cat-watcher logs [<agent>] [--follow] [--since DURATION_OR_ISO8601]
                 [--level LEVEL] [--camera NAME] [--grep SUBSTR]
                 [--json] [--config PATH]
```

- `<agent>` — positional, optional. Accepts `poller`, `alerts`, `web`, `backup`,
  `cli`. Omitted: read all four LaunchAgent files merged chronologically by
  `ts`. `cli.jsonl` is excluded by default to keep the view focused on services
  rather than command history; passing `cli` explicitly opts it in.
- `--follow` / `-f` — tail-and-watch loop. Poll selected files at 0.5s cadence
  using a remembered file-position-and-inode tuple per file so rotations are
  handled (re-open + replay any missed bytes). Standard SIGINT/EOF exits
  cleanly.
- `--since DURATION_OR_ISO8601` — accepts `30m`, `1h`, `7d` (regex
  `^\d+[smhd]$`) or any ISO 8601 the existing poller `_parse_iso_datetime`
  helper handles. Naive datetimes are OS-local (the same convention the poller
  uses).
- `--level LEVEL` — drop records below this level. Compare against
  `logging.getLevelName(record_level)`.
- `--camera NAME` — keep only records whose `extras.camera_name == NAME`.
  Records without that key are dropped.
- `--grep SUBSTR` — case-insensitive substring match on `msg`.
- `--json` — pass-through mode: emit raw JSONL lines unchanged. Useful for
  piping into `jq`. The pretty format is the default.
- `--config` — same as the rest of the umbrella CLI; resolves `internal_root` so
  the viewer knows which `logs/` to read.

Pretty format (default, when stdout is a TTY):

```text
2026-05-05 14:42:13  INFO   poller   ingested clip   camera=office clip_id=4217
2026-05-05 14:42:14  WARN   alerts   rule fired      rule=INACTIVITY camera=pantry
2026-05-05 14:42:15  ERROR  web      handler error   path=/clips/123  [+traceback]
```

- Timestamp rendered in OS-local time (the operator runs the viewer on the same
  host they read on).
- Columns: timestamp, level (right-padded to 5 chars; abbreviated `WARN` and
  `ERROR` to keep them aligned), agent, msg, extras as space-separated
  `key=value` pairs sorted by key.
- Color when `sys.stdout.isatty()`: DEBUG=dim, INFO=default, WARNING=yellow,
  ERROR=red, CRITICAL=red+bold. Use raw ANSI escape codes; no `rich` dep.
- Records with exception fields render with a trailing `[+traceback]` marker.
  The traceback itself is not expanded inline; a future `--show-traceback` flag
  would do that.

Help text on every flag (the existing umbrella CLI sub-commands are the shape to
match).

## Known limitations carried into Version 2

### Detector accuracy on top-down litter-box camera angles is unverified

Stock YOLOv11n is trained on COCO, where the "cat" class is dominated by natural
photography angles (face/body visible, normal pose). The cat-watcher cameras
point downward at a litter box and routinely capture cats in poses YOLO has seen
comparatively few of: from above, from behind, partially occluded by the
litter-box rim, curled up, or with the head buried. Spot-checking representative
frames against `yolo11n.pt` shows the model sometimes confidently misclassifies
a cat as a dog (COCO class 16). Real production accuracy on these specific
camera angles is unmeasured.

What this means for Version 1:

- The `INACTIVITY` and `FREQUENCY` alert rules use the same
  `COALESCE(manual_has_cat, has_cat)` projection. The `manual_has_cat` column is
  the operator's correction signal: every clip on `/clips/{id}` has a one-click
  label form (Task 22), and a corrected label retroactively flows through the
  alert rules and the `/stats` aggregations. So the system remains useful even
  if the detector's first-pass accuracy is mediocre — it just leans on the
  operator more than it would with a fine-tuned model.
- Default `detector.confidence_threshold = 0.35` may be too high for this angle;
  operators can lower it via `config.toml` once they have a feel for the
  false-negative rate on their specific camera placement.

What Version 2 should address:

- **Accuracy benchmark suite.** Once `manual_has_cat` has been populated for
  ~100 clips across both cameras, those become a real ground-truth set for
  measuring the model's precision/recall on production angles.
- **Fine-tuning vs. model swap.** With the labeled set above, the choice between
  fine-tuning yolo11n on operator clips vs. switching to a larger model
  (yolo11m/l) vs. a domain-specific model becomes empirical rather than guessed.
- **Integration test ground truth.** Bundle one or two real frames extracted
  from `videos/` (operator-owned; attribution + license is the operator's call)
  and switch the integration test from the synthesized `lavfi testsrc` source to
  those real frames, asserting realistic detection rather than just pipeline
  execution.
