# Cat Watcher — Phase 1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:subagent-driven-development (recommended) or
> superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the Phase 1 Cat Watcher system per
`docs/specs/2026-05-01-cat-watcher-design.md`: ingest motion clips from two
Amcrest cameras, classify cat-vs-no-cat, browse via web UI, alert on inactivity
/ frequency / agent failures, run unattended on a Mac mini.

**Architecture:** Single `cat_watcher` Python package, four LaunchAgents
(poller, alerts, web, backup) sharing one SQLite DB on internal storage and
clips/thumbs/backups on an external drive. Strict file-before-DB-row ordering;
preservation semantics on camera state; cross-agent heartbeat watchdog;
cool-down keyed on `(camera_id, alert_type)` for camera alerts and
`(NULL, alert_type)` for the seven non-camera alert types.

**Tech Stack:** Python 3.14, pixi (conda-forge for arch-pinned ML wheels + PyPI
for everything else), FastAPI + Jinja2 + htmx, SQLAlchemy 2.0 + Alembic, SQLite
(WAL), pydantic-settings, ultralytics YOLOv11n, ffmpeg, httpx,
ruff/basedpyright/mypy/pylint, AGPL-3.0.

> [!TIP]
> This project uses python 3.14. It does not need
> `from __future__ import annotations`

## How to Use This Plan

1. **Sequence is strict.** Tasks build on each other; do not reorder.
2. **Test Driven Development.** Use TDD best practices.
3. **Commit policy.** The user uses **signed git commits** (per CLAUDE.md). The
   implementing agent must NOT run `git commit` directly. Each task ends with a
   "Commit checkpoint" step that prepares the working tree (`git status` clean
   of untracked work, all tests passing) and tells the user the suggested commit
   message — the user runs the commit themselves.
4. **Dependency policy.** Never edit `pyproject.toml` to add/remove/upgrade
   dependencies by hand. Use the project's package manager CLI (per CLAUDE.md).
   The manager is `pixi`. PyPI deps are added with `pixi add --pypi <pkg>`
   (runtime) or `pixi add --pypi --feature dev <pkg>` (dev). Conda-forge deps
   (the arch-pinned ML stack) are added with `pixi add <pkg>` — no `--pypi`
   flag. Use `pixi remove [--pypi] [--feature dev] <pkg>` to drop. Never edit
   `pyproject.toml` dependency blocks by hand.
5. **Lint policy.** Linters are project standards. If a rule fires, the default
   is to comply, not silence (per CLAUDE.md). Silencing requires explicit user
   approval.
6. **Test doubles policy.** Per CLAUDE.md: no double > fake > stub > spy > mock.
   Mock only at boundaries owned by third parties (Amcrest API via respx, SMTP
   via aiosmtpd or `unittest.mock.patch('smtplib.SMTP')`, `osascript` via
   `subprocess.run` patch, YOLO model via
   `unittest.mock.patch('ultralytics.YOLO')`). Internal `cat_watcher.*` modules
   are real in tests.
7. **Phase checkpoints.** The plan is divided into 7 phases. After each phase,
   all tests pass and the system is in a consistent state — a natural place to
   pause for review.
8. **Reference inputs.** The `tmp-env-template/` directory contains tooling
   files copied from a sibling project. Reuse them verbatim where the spec says
   so (§10.2). The directory is deleted in Task 28.

## Resolutions for spec §13 open items

The spec defers five decisions to writing-plans. This plan resolves them as
follows:

1. **`amcrest_client` implementation** — `httpx.Client` with `httpx.DigestAuth`,
   not `python-amcrest`. Reasoning: (a) `python-amcrest` is licensed
   **GPL-2.0-only**, which is incompatible with the project's
   `AGPL-3.0-or-later` license per FSF compatibility rules; (b) it pulls
   `requests` + `urllib3` + `argcomplete` + sync/async stacks alongside `httpx`,
   which is heavy for a 2-camera 5-min cadence; (c) it does not yet declare
   Python 3.14 support (classifiers stop at 3.11). The API surface we need
   (`factory.create` → `findFile` → `findNextFile` + streaming download) is
   small enough to implement directly against the vendored
   `Amcrest-HTTP_API_V3.26.pdf`. (Task 14.)
2. **First-run model-cache strategy** — **eager** via the
   `cat-watcher fetch-models` sub-command. The operator runs it during install,
   before `pixi run install-agents`, so the first poller tick is offline-safe.
   (Task 25.)
3. **Initial Alembic migration** — autogenerated from the Task 10 models, then
   reviewed by hand. (Task 11.)
4. **Test fixture cat photo** — the Wikimedia Commons calico-kitten public
   domain image listed in Task 12, scaled below 30 KB; attribution recorded in
   `LICENSES.md`. (Task 12.)
5. **Cool-down key for non-camera alerts** —
   `(camera_id IS NULL,
   alert_type)`: one global cool-down per non-camera
   alert type. Applies to `POLLER_STUCK`, `WEB_DOWN`, `WEB_FLAPPING`,
   `ALERTS_STUCK`, `DISK_LOW`, `STORAGE_UNAVAILABLE`, `BACKUP_STALE`. (Task 18.)
6. **Camera local timezone (UTC conversion source)** — add an optional
   `cameras[].timezone` field to `config.toml`; if unset, fall back to
   `web.display_timezone` (the common case is one operator with all cameras and
   the host in the same zone). Final verification of which Amcrest API endpoint
   reports the camera's configured TZ happens during Task 14 implementation (the
   `Amcrest-HTTP_API_V3.26.pdf` is the reference); `pixi run test-cameras` then
   reports any drift between camera-reported and configured TZ at install time,
   and the operator sets `cameras[].timezone` explicitly if mismatched. (Tasks
   8, 14, 25.)

### Task 7a: Migrate from uv to pixi — retrospective

**Why:** PyPI dropped macOS-x86_64 wheels for the ML stack (torch, onnxruntime,
likely opencv-python in upcoming releases). Pixi sources from conda-forge, which
maintains osx-64 builds. This is the only workable path that doesn't require
hardware replacement or Docker on the Mac mini.

**What landed (deviates from spec §10.3 — keep this section as the source of
truth for the dependency split):**

- **Conda-forge** (in `[tool.pixi.dependencies]`) — only the packages with arch
  concerns: `python = "3.14.*"`, `pytorch`, `ultralytics`, `opencv`, `numpy`,
  `ffmpeg`. (Spec §10.3 also lists `pillow`; in practice it is pulled
  transitively and does not need to be pinned at the project root.)
- **PyPI runtime** (in `[project] dependencies`) — `alembic`, `fastapi`,
  `httpx`, `jinja2`, `pydantic-settings`, `python-dotenv`, `sqlalchemy`,
  `uvicorn[standard]`. Standard PEP 621 location.
- **PyPI dev** (in `[dependency-groups] dev`, PEP 735) — `aiosmtpd`,
  `basedpyright`, `mypy`, `pylint`, `pylint-per-file-ignores`,
  `pylint-pytest>=2.0.0a1`, `pylint-venv`, `pyproject-fmt`, `pytest`,
  `pytest-cov`, `pytest-mock`, `pytest-randomly`, `respx`, `ruff`. Pixi
  auto-maps the `dev` group to a feature; the `default` environment opts in.
- **Pixi-only PyPI block** (in `[tool.pixi.pypi-dependencies]`) — just the
  editable self-install: `cat-watcher = { path = ".", editable = true }`.
- **`[tool.pixi.workspace]`** — `channels = ["conda-forge"]`,
  `platforms = ["linux-64", "osx-64", "osx-arm64"]`.
- **`[tool.pixi.environments]`** —
  `default = { features = ["dev"],
  solve-group = "default" }`.

The split rule: PyPI for everything except packages whose macOS-x86_64 wheels
are missing on PyPI but available on conda-forge. Standard PEP 621 / PEP 735
locations are preferred over `[tool.pixi.feature.*.dependencies]` because pixi
already understands them and IDE / build tooling reads them natively.
**Therefore the `[project] dependencies` and `[dependency-groups]` blocks stay**
— they are the canonical source for everything the prod machine can fetch from
PyPI directly.

**Brewfile:** added `brew "pixi"`, removed `brew "uv"`. (`uv` was only useful
during the migration window and is gone now that the runtime is fully pixi.)

**Task runner:** `[tool.pixi.tasks]` replaces the `justfile`. All pixi tasks
listed in spec §10.4 are in place; `pixi task list` shows the full surface.

**`.gitignore`:** `.pixi/` excluded (per-machine cache). `pixi.lock` tracked.

### Task 9: Storage module

**Files:**

- Create: `src/cat_watcher/storage.py`
- Create: `tests/unit/test_storage.py`

**Implementation Requirements:**

- Implement layout management to ensure predictable persistence directories.
- Define exceptions: `StorageError` (base) and `StorageUnavailableError`
  (subclass) raised when a wait times out.
- `ensure_storage_layout(internal_root, storage_root)` generates required
  subfolders for internal operations (`models`, `logs`) and storage operations
  (`clips`, `thumbs`, `backups`). Loudly fails via `StorageError` if the root
  paths themselves do not exist (operator fixed a typo, didn't auto-create a
  wrong directory).
- `storage_available(path) -> bool` — true if `path` is an existing, writable
  directory (verified by writing then deleting a `tempfile.NamedTemporaryFile`
  inside it).
- `wait_for_storage(path, *, interval_seconds, timeout_seconds) -> None` — EAFP.
  Polls `storage_available(path)` every `interval_seconds`. Returns `None` on
  success. Raises `StorageUnavailableError` when the cumulative wait reaches
  `timeout_seconds`. Both keyword args default to the values in `[storage]`
  config (`wait_interval_seconds = 10`, `wait_timeout_seconds = 600`); tests
  override.
- Callers: `try: wait_for_storage(...)` then proceed;
  `except
  StorageUnavailableError:` log CRITICAL and exit non-zero.

**Test Requirements (`tests/unit/test_storage.py`):**

- `ensure_storage_layout` correctly generates the exact internal and storage
  subfolders.
- `ensure_storage_layout` is idempotent on repeated executions.
- `ensure_storage_layout` raises `StorageError` if the root paths are missing.
- `storage_available` for both writable and missing directory conditions.
- `wait_for_storage` returns `None` immediately when the path is available.
- `wait_for_storage` raises `StorageUnavailableError` on timeout (mock
  `time.monotonic` and `time.sleep` so the test runs in milliseconds).

### Task 13: Notifier — Email + macOS Notifications

**Files:**

- `src/cat_watcher/notifier.py`
- `tests/unit/test_notifier.py`

#### Implementation Requirements (`notifier.py`)

- **Data Structures:** Define `EmailResult` and `NotifResult` frozen dataclasses
  containing a boolean `ok` status and an optional `error` string.
- **Email Functionality:** Implement
  `send_email(subject: str, body: str, *, secrets: EmailSecrets, rules:
  EmailRulesConfig) -> EmailResult`.
  Both config objects passed as parameters (no module-level config import —
  keeps the function pure and trivially testable). Caller (alerts agent / poller
  dispatch) loads config once and threads the relevant pieces in.
  - Must use **SMTP STARTTLS** and login via credentials in `secrets`.
  - Must construct a standard `EmailMessage` with appropriate headers (Subject,
    From: `secrets.gmail_user`, To: `secrets.alert_to_addresses`, SMTP host/port
    from `rules.smtp_host` / `rules.smtp_port`).
  - When `rules.enabled is False`, return
    `EmailResult(ok=True,
    error="disabled")` without touching SMTP.
    (Disabled is a valid final state, not a failure.)
  - Must handle `OSError` and `smtplib.SMTPException`, logging the failure and
    returning a failed `EmailResult` instead of raising.
- **macOS Notification Functionality:** Implement
  `send_macos_notification(title: str, body: str, *, rules: MacOsRulesConfig)
  -> NotifResult`
  — config passed as parameter, same rationale as above.
  - When `rules.enabled is False`, return
    `NotifResult(ok=True,
    error="disabled")` without invoking `osascript`.
  - Must invoke the system `osascript` via `subprocess.run`.
  - Must escape double quotes in the title and body to prevent script injection.
  - Must return a failed `NotifResult` if the process returns a non-zero exit
    code or if the executable is missing.

#### Test Requirements (`test_notifier.py`)

- **SMTP Mocking:** Patch `smtplib.SMTP` to verify that `starttls()`, `login()`,
  and `sendmail()` are called with the correct parameters from the
  configuration.
- **Email Error Handling:** Verify that the function returns a failure result
  when `smtplib` raises a connection error.
- **Subprocess Mocking:** Patch `subprocess.run` to verify that the `osascript`
  command is formatted correctly with the `display notification` syntax.
- **Notification Robustness:** Ensure the notifier does not crash if the
  notification process fails or times out.

### Task 14: Amcrest Client — Lazy Iteration and Streaming

**Files:**

- `src/cat_watcher/amcrest_client.py`
- `tests/unit/test_amcrest_client.py`

#### Implementation Requirements (`amcrest_client.py`)

- **Client Architecture:** Create an `AmcrestClient` class utilizing
  `httpx.Client` with `httpx.DigestAuth`. Constructor takes a `CameraConfig`
  (per-camera host/port/credentials/timezone).
- **Exception Hierarchy:** Define custom exceptions: `CameraUnreachable`
  (network failure), `CameraAuthError` (401/403), and `CameraAPIError`
  (everything else, including malformed responses).
- **Recording Metadata:** Define a `Recording` frozen dataclass with fields:
  `source_filename: str` (matches `clips.source_filename` schema column),
  `camera_path: str` (vendor URL fragment for download), `start_ts: datetime`
  (UTC, tz-aware), `end_ts: datetime` (UTC, tz-aware), `file_size_bytes: int`
  (matches `clips.file_size_bytes`).
- **Lazy Pagination (`iter_recordings`):**
  - Implement the session-based search flow: `factory.create` → `findFile` →
    `findNextFile` → `close`/`destroy`. Use a `try`/`finally` so the search
    handle is always destroyed even on exception mid-iteration.
  - Use a generator to yield `Recording` objects as they are parsed from the
    camera's plain-text response format.
  - **Camera-local timezone source:** use
    `ZoneInfo(camera_config.timezone or app_config.web.display_timezone)` when
    converting filename / start / end strings to UTC. The fallback is documented
    in the §13 resolution #6.
- **Streaming Download (`download_recording`):**
  - Use `httpx.stream("GET", url)` and `iter_bytes()` to pipe bytes into a
    `.part` file open with `O_WRONLY | O_CREAT | O_TRUNC`. After the loop,
    `os.fsync(fp.fileno())` before `close()` so the bytes are durable.
  - Atomic `os.replace(part_path, final_path)` on successful completion only.
  - Leave the `.part` behind on any exception so a manual sweep / retention pass
    can detect the partial.
- **Retry policy (per spec §4.2 — applies to BOTH `iter_recordings` and
  `download_recording`):**
  - **3 attempts total**, **10-second delay** between attempts, on
    `httpx.ConnectError`, `httpx.ReadTimeout`, `httpx.RemoteProtocolError`, or
    5xx responses. After exhausting retries, raise `CameraUnreachable`.
  - **No retry on 4xx** — those are config issues (auth, bad path). Immediately
    raise `CameraAuthError` (401/403) or `CameraAPIError` (others).
  - The retry helper is internal; the public methods raise the typed exception
    when retries are exhausted, so callers (poller) catch a single exception
    type per failure mode.

### Task 16: Retention Sweep

**Files:**

- `src/cat_watcher/retention.py`
- `tests/unit/test_retention.py`

#### Implementation Requirements

- **Data Purge Logic:** `sweep(*, retention: RetentionConfig, now:
  datetime)`
  — durations come from `[retention]` config (`clip_days`, `agent_starts_days`,
  `alerts_sent_days`, all default 30 per Task 8). Tests pass an explicit `now`
  for determinism.
- **Two-pass clip sweep (per spec §4.4 step 7):** the clip portion of the sweep
  runs in two distinct passes that the test suite exercises independently:
  1. **DB-driven pass.** For each clip with
     `start_ts < now -
     retention.clip_days`: delete the row inside a
     transaction first, then unlink the clip and thumbnail files. Row-first
     ordering means a crash between the two leaves only an orphan file, never a
     dangling row pointing at a missing file.
  2. **Orphan sweep.** Scan `clips/` and `thumbs/` for files whose mtime is
     older than `now - retention.clip_days` and that have no matching `Clip`
     row, and unlink them. This cleans the orphans pass 1 leaves behind on
     crash, plus any other stray files (e.g., a partially-written download from
     before the file-before-row ordering existed).
     - **Efficiency note:** load the surviving `Clip.file_path` and
       `Clip.thumb_path` values into a single in-memory `set` once, then walk
       the filesystem and check membership. Don't issue one DB query per file —
       the sweep should be O(N + M), not O(N × M).
  3. **Empty date directory cleanup.** After both passes, walk `clips/<slug>/`
     and `thumbs/<slug>/`; remove any date subdirectory that is empty
     (`os.rmdir`). Otherwise empty `2024-01-15/` dirs accumulate forever even
     though the spec promised "delete entire date directories at once" (§6.1).
- **Other tables:**
  - Prune `agent_starts` rows older than `now - retention.agent_starts_days`
    (matches the visibility horizon the `WEB_FLAPPING` watchdog and the forensic
    "when-did-X-restart" UI rely on).
  - Prune `alerts_sent` rows older than `now - retention.alerts_sent_days`
    (matches `/alerts` route's "last N days" claim — the route reads the same
    config value).
- **Resilience:** Handle `OSError` exceptions during file deletion or directory
  cleanup (file already missing, dir not empty due to race) without crashing the
  sweep — log WARNING and continue.
- **Reporting:** Return a structured `RetentionReport` dataclass with counts:
  `clips_removed_pass1`, `orphans_removed_pass2`, `dirs_removed`,
  `agent_starts_pruned`, `alerts_sent_pruned`.

#### Test Requirements

- **Pass 1 (functional sweep):** Records and files older than the cutoff are
  removed; "younger" data remains intact.
- **Pass 2 (orphan sweep):** Pre-seed an orphan file (no matching Clip row) with
  old mtime; verify it is unlinked. Pre-seed a young orphan file; verify it is
  preserved.
- **Crash-tolerance shape:** Simulate a pass 1 crash between row delete and file
  unlink; verify the next sweep's pass 2 cleans up the orphan.
- **Empty-dir cleanup:** seed a date directory whose only files are removed by
  pass 1; verify the directory is `rmdir`-ed in pass 3. Seed a date directory
  with one young file remaining; verify the directory is kept.
- **Configurability:** with `retention.clip_days = 7`, files older than 7 days
  are removed; with the default 30, only files older than 30 days are removed.
- **Idempotency:** Running the sweep twice in a row results in zero additional
  deletions and no errors.
- **Integrity:** File deletion failures are logged as warnings but do not halt
  the process.
- **agent_starts / alerts_sent pruning:** Old rows in both tables are removed;
  recent rows are preserved. Independent retention configs (e.g.,
  `agent_starts_days = 7` with `alerts_sent_days = 30`) honored.

### Task 17: Poller Agent

**Files:**

- `src/cat_watcher/poller.py`
- `tests/unit/test_poller.py`
- `tests/integration/test_poller_end_to_end.py`

#### Implementation Requirements

- **Concurrency Control:** Implement an exclusive, non-blocking PID file lock on
  `<internal_root>/.poller.pid` (`fcntl.flock` with `LOCK_EX | LOCK_NB`). If
  another poller process holds the lock, exit cleanly with code 0 (per spec §4.4
  step 1 — covers manual `pixi run poll-once` overlapping with a LaunchAgent
  tick).
- **Environment Validation:** Use the §4.13 storage-availability wait
  (10s/10min) before proceeding; the poller writes to the external drive so it
  cannot run without it. Insert an `agent_starts(agent_name='poller', ...)` row
  before the per-camera loop.
- **Ingestion Pipeline:**
  - Connect to Amcrest cameras via the client.
  - Iterate through available recordings since the last successful poll
    (`last = camera.last_polled_at or now - retention.clip_days days`). The
    "first-run backfill" matches the retention window so a fresh install sees
    the same horizon the steady-state system maintains.
  - **First-run scoping (per spec §11.1):** to avoid pulling the full backfill
    on a fresh install, the operator can run
    `pixi run poll-once --since <recent-iso8601>` (optionally `--no-detect`)
    BEFORE `pixi run install-agents`. That seeds `cameras.last_polled_at`, so
    the LaunchAgent's first scheduled tick picks up from there.
  - **Processing Steps:** Download video → Generate thumbnail via FFmpeg → Run
    AI detection (if enabled) → `fsync` both files (download already fsyncs per
    Task 14; thumbnail must do the same) → **only then** insert the `clips` row.
    This strict file-before-row ordering guarantees the web UI never sees a row
    pointing at a partial file.
  - Idempotency: rely on `UNIQUE (camera_id, source_filename)` to skip
    recordings already in the DB.
- **Camera state — preservation semantics (per spec §4.4 step 5):** the update
  step does NOT blindly write all fields. Each field follows its own rule:
  - `last_polled_at` — set to `now` on every successful tick.
  - `last_clip_at` — set to `max(start_ts of new clips)` only when clips were
    actually ingested this tick; otherwise preserve the previous value.
  - `last_cat_seen_at` — set to
    `max(start_ts where COALESCE(manual_has_cat,
    has_cat) = true)` only when
    at least one new clip is cat-positive; otherwise preserve.
  - `poll_status` — set to `ok` on success; on failure, set the appropriate
    non-`ok` status (`unreachable` / `error`).
  - `poll_status_since` — set to `now` **only on the transition** from `ok` to
    non-`ok`. Cleared (NULL) on any transition back to `ok`. Left unchanged
    while still non-`ok`.
- **Error Isolation:** A crash or connection failure on one camera must not
  prevent the poller from attempting subsequent cameras.
- **Retention sweep:** call `cat_watcher.retention.sweep(...)` at end of tick
  (Task 16's two-pass implementation, with config from `[retention]`).
- **Heartbeat write:** at end of tick (after the per-camera loop and the
  retention sweep), upsert the `heartbeats` row for `agent_name='poller'` with
  `last_seen_at=now`. Per spec §4.4 step 6.
- **Watchdog (`ALERTS_STUCK`):** Read the `alerts` heartbeat. If older than
  `[alerts].alerts_stuck_minutes` (default 30 min), call
  `cat_watcher.alerts.dispatch_alert("ALERTS_STUCK", camera_id=None, ...)` — see
  Task 18 for the dispatch helper. The helper handles cool-down + the
  `alerts_sent` audit row + email/macOS sender invocation. The poller does NOT
  manage cool-down state itself.
- **`analysis_error` field semantics:** the column doubles as "why this clip is
  un-analyzed" — both genuine ffmpeg/detector failures (e.g.,
  `"ffmpeg returned 1: <stderr tail>"`) and explicit `--no-detect` skips
  (`"skipped: --no-detect"`). The web UI's "un-analyzed" filter
  (`analysis_error IS NOT NULL`) groups them. If a future need arises to
  distinguish, add a separate `analysis_skipped: bool` column then.
- **CLI flags for `cat-watcher-poller --once` (per spec §4.10):**
  - `--config PATH` — config-path override (precedence: CLI > env > default).
  - `--camera NAME` — restrict to a single configured camera (default: all).
  - `--since YYYY-MM-DDTHH:MM:SS` — override per-camera `last_polled_at`.
  - `--until YYYY-MM-DDTHH:MM:SS` — upper bound on the search window (default:
    now).
  - `--limit N` — cap clips downloaded this invocation.
  - `--no-detect` — skip detection; rows insert with
    `analysis_error = "skipped: --no-detect"`, `has_cat = false`, and the YOLO
    model is never loaded (so a fresh dev clone needs no model weights).
  - `--list-only` — print what would be fetched; no downloads, no DB writes.

#### Test Requirements

- **Unit Tests:**
  - PID file locking prevents secondary process execution (second invocation
    exits 0; no duplicate downloads, no DB write conflicts).
  - Camera state advances correctly when new clips are found.
  - **Heartbeat write:** verify the `heartbeats` row for `agent_name='poller'`
    is upserted with `last_seen_at=now` at end of tick — including on a tick
    where 0 new clips were found.
  - **Preservation semantics:** with no new clips ingested, `last_clip_at` /
    `last_cat_seen_at` are NOT overwritten.
  - **Preservation semantics:** with new clips but no cat-positive clip,
    `last_cat_seen_at` is NOT overwritten.
  - **`poll_status_since` transitions:** ok → unreachable sets it; second
    consecutive unreachable tick leaves it unchanged; unreachable → ok clears it

  - "unreachable" / "error" statuses are correctly recorded on connection

  - `ALERTS_STUCK` fires when the alerts heartbeat is older than

  - `--list-only` writes nothing to disk or DB.

  - Execute a full end-to-end "tick" using a mocked Amcrest API (`respx`) and

  - Verify clips are recorded in the database and files land in the correct

### Task 18: Alerts Agent

**Files:**

- `src/cat_watcher/alerts.py` (rule evaluation + dispatch entry point)
- `src/cat_watcher/alert_templates.py` (subject/body renderers per spec §4.14)
- `tests/unit/test_alerts.py`
- `tests/unit/test_alert_templates.py`

#### Module split rationale

Three distinct concerns live in three places to keep imports one-way and
testability high:

- `cat_watcher.notifier` — pure outbound senders (Task 13). Knows nothing about
  alert types, cool-down, or DB state.
- `cat_watcher.alert_templates` — pure renderers. One function per alert type
  (`render_inactivity(camera, branch, ...) -> AlertContent`,
  `render_frequency(camera, count, ...)`, etc.). Returns an `AlertContent`
  dataclass with `subject`, `body`, `macos_summary` (≤120 chars). No I/O, no
  config, no DB. Trivially testable as snapshot tests.
- `cat_watcher.alerts` — rule evaluators + the shared `dispatch_alert(...)`
  helper. The poller imports `dispatch_alert` for `ALERTS_STUCK` (Task 17).
  Imports flow `poller → alerts → {notifier, alert_templates}`; nothing imports
  `poller`, so no circular import risk.

#### Shared dispatch helper

`dispatch_alert(alert_type: AlertType, *, camera_id: int | None,
content: AlertContent, secrets: EmailSecrets, rules: AlertConfig,
session: Session) -> None`

Behavior:

1. Compute the cool-down key: `(camera_id, alert_type)` for camera alerts;
   `(NULL, alert_type)` for the seven non-camera types.
2. Look up the most recent `alerts_sent` row matching the key. If
   `now - sent_at < cooldown_for(alert_type, rules)`, **log at INFO** with the
   suppression reason and return — **do NOT write a suppressed row to
   `alerts_sent`** (otherwise an active 24h-cool-down alert would generate ~96
   rows/day per type). Spec §6.2 has no `suppressed` column, by design.
3. Otherwise call `notifier.send_email(...)` and
   `notifier.send_macos_notification(...)` (both honor their `enabled` flags).
   Write a single `alerts_sent` row capturing `email_ok` / `macos_ok` /
   `delivery_error`.

`cooldown_for(alert_type, rules)` returns `_HARDCODED_COOLDOWN_OVERRIDES`'s
value if the type is in the dict (`WEB_FLAPPING=1h`, `DISK_LOW=24h`,
`BACKUP_STALE=24h`); otherwise `rules.cooldown_hours` (default 6h). The
overrides live in a module-level dict in `cat_watcher.alerts` with a comment
explaining why they are intentionally not in `config.toml` (per spec §4.5: a
flap detector with a 6h cool-down would miss the next flap; a once-per-24h
backup's staleness check would fire repeatedly within a single missed run if the
cool-down were hours-scale). One-line edit if the operator needs to change them;
no config schema churn.

#### Implementation Requirements

- **Startup:** Insert an `agent_starts(agent_name='alerts', ...)` row. **Skip
  the §4.13 storage wait** — the alerts agent's state lives on internal storage,
  so it must run even when the external drive is offline (so it can fire
  `STORAGE_UNAVAILABLE`). Each tick performs its own one-shot probe of
  `storage_root` instead.
- **Per-camera rules:**
  - **`INACTIVITY`** — fires if any branch is true:
    - `last_cat_seen_at` not NULL and
      `now - last_cat_seen_at > inactivity_hours` (default 12h). Note
      `last_cat_seen_at` updates from the `COALESCE(manual_has_cat, has_cat)`
      projection so manual labels feed back in.
    - `last_clip_at` not NULL and `now - last_clip_at > inactivity_hours`
      (camera producing nothing — silent-SD-failure case).
    - `poll_status != 'ok'` — fires immediately, no time threshold; cool-down
      still applies.
    - The alert body distinguishes which branch fired (no cats / no clips /
      camera unreachable since `poll_status_since`).
    - If both `last_cat_seen_at` and `last_clip_at` are NULL the rule does not
      fire (no baseline yet — backfill on first poll covers this).
  - **`FREQUENCY`** — fires when
    `count(clips where camera_id=cam AND COALESCE(manual_has_cat, has_cat)=true
    AND start_ts > now - frequency_window_hours) >= frequency_threshold_count`
    (defaults: 6h window, 8 cat-positive clips). The
    `COALESCE(manual_has_cat, has_cat)` form means the rule respects user
    corrections.
- **Cross-agent watchdogs (heartbeat-based, all thresholds in `[alerts]`):**
  - **`POLLER_STUCK`** if poller heartbeat older than `poller_stuck_minutes`
    (default 15).
  - **`WEB_DOWN`** if web heartbeat older than `web_down_minutes` (default 5).
  - **`WEB_FLAPPING`** if `≥web_flapping_threshold_count` rows (default 5) in
    `agent_starts` for `agent_name='web'` within the last
    `web_flapping_window_minutes` (default 30). Body includes the last 50 lines
    of `web.stderr.log`.
  - (`ALERTS_STUCK` is fired by the **poller**, not here — see Task 17.
    Threshold is `[alerts].alerts_stuck_minutes`, default 30.)
- **Storage rules (every tick, even when external drive is offline):**
  - **`STORAGE_UNAVAILABLE`** if the alerts agent's own write probe to
    `storage_root` fails (`tempfile.NamedTemporaryFile(dir=storage_root)`).
    Recorded in `alerts_sent` like any other alert (DB lives on internal
    storage). No marker-file workaround required.
  - **`DISK_LOW`** if
    `shutil.disk_usage(storage_root).free / total <
    [alerts].disk_low_threshold_fraction`
    (default 0.10). **Skip when `storage_root` is not mounted —
    `STORAGE_UNAVAILABLE` covers that case.**
  - **`BACKUP_STALE`** if no file matching `backups/cat_watcher-*.sqlite` has an
    mtime within the last `[alerts].backup_stale_hours` (default 36). **Skip
    when the most recent `alerts_sent` row for `STORAGE_UNAVAILABLE` is newer
    than its cool-down window** — this suppresses double-alerting on the same
    root cause.
- **Dispatch & cool-down:** all routing goes through the shared
  `dispatch_alert(...)` helper described above. Rule evaluators do not call
  `notifier` directly and do not write to `alerts_sent` directly. Suppressed
  events log at INFO; only dispatched events write a row.
- **Heartbeat:** Write the `alerts` heartbeat at end of tick.

### Task 19: Backup Agent

**Files:**

- `src/cat_watcher/backup.py`
- `tests/unit/test_backup.py`

#### Implementation Requirements

- **Startup:** Perform the §4.13 storage-availability wait (10s/10min — the
  backup target is on the external drive). Insert
  `agent_starts(agent_name='backup', ...)` after the wait succeeds. Then perform
  the backup + prune.
- **Hot Backup:** Use SQLite's online backup API (`sqlite3.Connection.backup()`)
  to copy `<internal_root>/cat_watcher.sqlite` to
  `<storage_root>/backups/cat_watcher-YYYY-MM-DD.sqlite`. Safe under WAL mode
  without blocking writers (per spec §4.12). Cross-volume direction is
  intentional — DB on internal, backup on external.
  - Open a **new** `sqlite3.connect(...)` connection to the source for the
    backup operation (not the SQLAlchemy session pool). The online backup API
    needs its own dedicated connection so it can iterate page-by-page without
    contending with SQLAlchemy's pooled handles.
- **Naming convention:** ISO-8601 dated filenames
  (`cat_watcher-2026-05-01.sqlite`).
- **Retention:** Keep the most recent `[backup].keep_count` daily backups
  (default 7); older files pruned by mtime each run.
- **Heartbeat exclusion:** The backup agent does NOT participate in the
  `heartbeats` watchdog scheme — its once-daily cadence would always look stale.
  Backup health is monitored via the `BACKUP_STALE` rule in Task 18 (filesystem
  mtime, not heartbeat).
- **Out of scope for backup:** clips and thumbnails — they're large, the

  territory.

#### Test Requirements

- **Data Integrity:** The produced backup file is a valid SQLite database
  containing the same rows as the source.
- **Pruning Logic:** Seed a directory with `keep_count + 3` dummy backup files
  of varying mtimes; verify only the `keep_count` most recent remain after the
  agent runs. Repeat with `keep_count = 3` to confirm the value is honored.
- **Storage wait:** with `storage_root` unmounted, the agent retries per the

- **agent_starts row:** verify a row is inserted on each successful run and is

### Task 23: Web Routes — Timeline

Scope: visual activity overview rendered as inline SVG with HTMX-driven range
presets.

#### Requirements & Instructions

- **Timeline Logic**:
  - Surface: `GET /` and `GET /timeline`.
  - Instruction: Support a `range` query parameter (`6h`, `24h`, `7d`, `30d`).
    Calculate the start window based on the current time.
  - Data Retrieval: Fetch all clips and alerts within the time window, grouped
    by camera.
- **SVG Rendering**:
  - Requirement: Generate a server-side SVG representing activity lanes for each
    camera.
  - Instruction: Represent clips as rectangles, width proportional to
    `duration_seconds`, color-coded by detection status:
    - green: `COALESCE(manual_has_cat, has_cat) = true`
    - gray: `COALESCE(manual_has_cat, has_cat) = false`
    - red outline: `analysis_error IS NOT NULL` (un-analyzed)
    - blue outline: `manual_has_cat IS NOT NULL` (human-labeled)
  - Represent alerts as vertical dashed lines spanning all lanes at each
    `alerts_sent.sent_at`, color-coded by `alert_type`, with a small label tag.
- **Density bucketing for windows > 24h (per spec §4.7.1):**
  - For ranges `7d` and `30d`, marker count per lane can exceed pixel-per-clip
    resolution. Cluster markers into per-hour bins per lane; render each bin as
    a heatmap-style cell whose color intensity encodes count (and includes the
    count as a `<title>` for hover).
  - For ranges `6h` and `24h`, render per-clip markers as above (no bucketing).
  - The threshold (24h) is hardcoded — the spec is explicit that bucketing kicks
    in based on rendering resolution, not user choice.
- **Interactivity**:
  - Use HTMX to swap the timeline content when switching range presets without a
    full page reload.
  - Implement a lightweight vanilla JavaScript hover-preview to display clip
    timestamps and detection scores when the mouse is over an SVG element.
- **Storage-offline degradation (per spec §4.13):**
  - The timeline renders inline `<img>` thumbnails sourced from
    `/media/thumb/{id}`. When those routes return 503, an `onerror` handler on
    the `<img>` swaps in a single bundled SVG placeholder so the timeline still
    reads as a structured event list rather than a wall of broken-image glyphs.
  - The handler running the request also performs a one-shot probe of
    `storage_root`. When the probe fails, the template renders a thin banner at
    the top of the page: "External storage offline — clip playback and
    thumbnails unavailable". When the probe succeeds, no banner.

#### Testing Requirements (`tests/integration/test_web_timeline.py`)

- **Render Check**: Verify the presence of `<svg>` and camera display names on

- **Parameter Handling**: Verify that the `range` parameter correctly adjusts

- **Per-clip rendering at ≤24h:** seed 10 clips spread over 6 hours; request

- **Density bucketing at >24h:** seed 50 clips spread over 7 days; request

### Task 26: LaunchAgent Infrastructure & Lifecycle Scripts

**Files:**

- `scripts/plists/com.robgant.cat-watcher.{poller,alerts,web,backup}.plist.template`
- `scripts/install_launchagents.sh`

#### 1. Plist Template Requirements

Each template must define a macOS LaunchAgent using the following configuration
surfaces to ensure consistent behavior and observability:

- **Execution Identity:** A unique `Label` following the
  `com.robgant.cat-watcher.[service]` convention.
- **Context:** Set `WorkingDirectory` to the repository root and provide the
  `CAT_WATCHER_CONFIG` environment variable.
- **Pathing:** Use `__REPO_DIR__` and `__INTERNAL_ROOT__` placeholders for
  dynamic substitution during installation.
- **Logging:** Redirect `StandardOutPath` and `StandardErrorPath` to
  `<internal_root>/logs/<agent>.{stdout,stderr}.log` (per spec §6.1) so logs
  remain accessible even when the external drive is offline.
- **`RunAtLoad`:** Every plist sets `<key>RunAtLoad</key><true/>`. Without this,
  periodic agents (`StartInterval`-based) do not fire on bootstrap until the
  next interval rolls over, which delays observability of any install issue.
- **Service Specifics — cadences read from `config.toml` at install time (the
  install script renders the templated plist with values from `[poller]`,
  `[alerts]`, `[backup]`):**
  - **Poller:**
    `<key>StartInterval</key><integer>{{ poller.cadence_seconds }}</integer>`
    (default 300 = 5 min).
  - **Alerts:**
    `<key>StartInterval</key><integer>{{ alerts.cadence_seconds }}</integer>`
    (default 900 = 15 min).
  - **Web:** Long-running process with `KeepAlive` (restart on non-zero exit)
    and `<key>ThrottleInterval</key><integer>30</integer>` to prevent
    crash-looping (per spec §8.2).
  - **Backup:** `StartCalendarInterval` with `Hour={{ backup.cadence_hour }}`
    and `Minute={{ backup.cadence_minute }}` (defaults 3, 0 — 03:00 local).

#### 2. Installation & Uninstallation Logic

The shell scripts manage the transition from templates to active system
services.

- **`install_launchagents.sh`:**
  - **Cadence injection from config:** invoke
    `pixi run python -m cat_watcher.scripts.render_plists` (a small helper) to
    read `config.toml` and emit the four final plist files into a temp dir. The
    helper does the templating in Python (where the config is already parsed)
    rather than asking the shell to parse TOML. The script then `cp`'s the
    rendered plists into `~/LaunchAgents/`.
  - **Placeholder injection:** `__REPO_DIR__` and `__INTERNAL_ROOT__`
    placeholders are replaced by the Python helper at the same time it writes
    the cadence values.
  - **Idempotency:** The script must handle existing agents by performing a
    `bootout` (if loaded) before bootstrapping the new plist.
  - **Directory Management:** Ensure the target log and `LaunchAgents`
    directories exist before deployment.
- **`uninstall_launchagents.sh`:**
  - **Cleanup:** Gracefully stop services using `launchctl` and remove the
    generated `.plist` files from the user's `LaunchAgents` directory.

#### 3. Validation

  stylistic consistency.
- **Verification:** Manual verification on the Mac mini host is required to
  ensure `launchctl` correctly interprets the bootstrapped plists.
